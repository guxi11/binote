# Session Handoff 协议 — 首次实验结论 (2026-07-10)

## 协议

把长会话在相位边界蒸馏为:**handoff brief**(常驻新会话的精度脊柱:宪章 + 检索地图 + 环境硬约束 + 未决事项)+ 细节沉入 `.binote/` 笔记(由 feature 001 Phase 1 的混合搜索按需召回)。新会话 = 干净上下文 + brief + binote 检索工具。

## 实验设置

本会话(feature 001 Phase 1 落地,~130k tokens)蒸馏为 [[_handoff/2026-07-10-feature-001-phase1.md]];干净子代理仅持 brief 路径,执行:6 题事实盲测(标注来源)、20 条声明的镜像笔记微审计、1 次混合搜索冒烟(需自行从 brief 推断正确验证方式)。

## 结果

- **盲测 6/6 全对**,来源标注准确;其中"MCP server 跑旧代码"这类会话态断言,接力者用发版序推理 + 空结果实测独立补证——超出 brief 给的信息。
- **审计 18/20 属实**,且反向抓出 brief 自身两处裂纹:EMBEDDINGS_VERSION"独立"措辞不准(实际三重校验)、模型体积三处数字打架(144/100/129MB)。
- **冒烟正确**:从 brief 推出"不能用 MCP 工具、要用 dist CLI",并以 `via` 字段 + RRF 签名分值 0.016 判定语义层生效。
- 成本:接力会话 56k tokens / 12 次工具调用,对照原会话 130k+ 仍在膨胀。
- **RAG 绑定自动成立**:brief 写入 `.binote/` 后即被混合搜索索引(冒烟第 2 名命中即 brief 本身),无需独立 session vector store。

## 关键教训(若做成 `/binote:handoff` skill,进蒸馏 prompt)

1. **运行时断言必须带机械锚点**:dist 新鲜度给 commit hash 核对命令;进程状态改写成会话无关的表述(如"发布版 ≤X 均无该功能")。
2. **数字给实测单值**,多文档互引时不许"约数"打架。
3. **冒烟给预期基线**(耗时量级、应命中什么、缓存应有几条),否则接力者无法区分"坏了"和"本来如此"。
4. 检索地图要标注已知悬空链接。
5. 失效模式全部集中在"不可机械核对的断言"上;蒸馏质量决定一切的预判成立,但结构(脊柱+召回网)本身零失误。

## 判定

协议可行,值得立项为 `/binote:handoff` skill;核心资产是蒸馏 prompt(上述 5 条即其骨架),存储与检索均复用现有设施。
